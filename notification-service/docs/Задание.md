Задание 1: Разработка распределенного сервиса отправки уведомлений на разные источники 

Описание: Необходимо разработать распределенный сервис для отправки уведомлений через различные каналы (e-mail, SMS, push-уведомления, мессенджеры и др.). Сервис должен быть масштабируемым и поддерживать гибкость в добавлении новых каналов. 

Цель: Создать сервис, который сможет отправлять уведомления в реальном времени на различные источники, обеспечивая при этом высокую надежность доставки и поддержку нескольких типов сообщений (текстовые, мультимедийные и т.д.). 

Задачи: 
1. Архитектура системы: 
    Спроектировать микросервисную архитектуру, которая включает в себя следующие компоненты: 
        - Шлюз уведомлений (Notification Gateway). 
        - Микросервисы для каждого канала отправки (Email Service, SMS Service, Push Service, и т.д.). 
        - Очереди сообщений для обработки задач асинхронно. 
        - Базу данных для хранения состояния отправленных уведомлений. 

2. Шлюз уведомлений (Notification Gateway): 
    - Принимает входящие запросы на отправку уведомлений от других систем или сервисов. 
    - Маршрутизирует уведомления в соответствующие микросервисы-отправители (Email Service, SMS Service и т.д.). 
    - Обеспечивает повторную попытку отправки уведомлений в случае ошибок. 
    
3. Микросервисы каналов (Email, SMS, Push и т.д.): 
    - Каждый микросервис отвечает за отправку уведомлений через конкретный канал. 
    - Должен поддерживать возможность расширения для добавления новых каналов. 
    - Микросервисы должны быть независимыми и масштабируемыми. 

4. Очереди сообщений: 
    - Использовать очередь сообщений (например, RabbitMQ, Kafka) для асинхронной обработки уведомлений. 
    - Очереди должны поддерживать повторную отправку в случае сбоя при обработке. 
    
5. Управление состоянием: 
    - Хранить состояние отправленных уведомлений: отправлено успешно, ошибка, повторная отправка. 
    - Логировать все попытки отправки уведомлений, успешные и неудачные, для возможности аудита. 

6. API: 
    - Предоставить REST API для интеграции с другими сервисами, позволяющее отправлять уведомления через различные каналы. 
    - API должно поддерживать параметры: тип уведомления (e-mail, SMS, push), получатель, сообщение и другие метаданные. 

7. Конфигурация и расширяемость: 
    - Возможность легко добавлять новые каналы отправки (например, интеграция с мессенджерами или социальными сетями). 
    - Управление конфигурациями отправки (например, API-ключи для сторонних сервисов отправки, настройки ретраев и т.д.). 

8. Мониторинг и логирование: 
    - Реализовать систему мониторинга отправки уведомлений (с использованием Prometheus или других решений). 
    - Настроить логирование и уведомления для случаев сбоя отправки сообщений. 

9. Тестирование: 
    - Написать автотесты для каждого микросервиса и шлюза уведомлений.
    - Обеспечить нагрузочное тестирование для оценки масштабируемости системы. 
    
Рекомендуемые технические требования: 
    Язык программирования: C# (.NET Core). 
    Архитектура: микросервисная. 
    Брокер сообщений: RabbitMQ или Kafka. 
    База данных: PostgreSQL или MongoDB. 
    Мониторинг: Prometheus, Grafana. 
    Логирование: ELK Stack (Elasticsearch, Logstash, Kibana). 

Критерии оценки: 
    Сервис должен быть развернут в контейнерах (Docker). 
    Микросервисы должны быть масштабируемы и устойчивы к сбоям. 
    Уведомления должны корректно отправляться через все каналы с возможностью добавления новых. 
    Реализована поддержка повторов и логирование всех операций.
